## ❓ 자주 하는 질문

### Q1. 왜 EC2가 2개 필요한가요?
```
현실 세계를 시뮬레이션하기 위해서입니다.
• 개발자가 모델을 만드는 환경 (EC2 #1)
• 실제로 서비스하는 서버 (EC2 #2)

실무에서는 이 두 환경이 완전히 분리되어 있습니다.
EC2 #1 → ECR → EC2 #2 흐름이 바로 CI/CD 파이프라인의 기본입니다.
```

### Q2. Triton 대신 Flask로 만들면 안 되나요?
```
가능하지만, 직접 구현해야 할 것이 많습니다:
• 동시 요청 처리 (멀티스레딩/비동기)
• GPU 메모리 관리
• 모델 로딩/언로딩
• 배치 처리 최적화
• 모니터링/메트릭

Triton은 이런 것들을 이미 다 해결해 놓았습니다.
```

### Q3. Docker 없이 그냥 서버에 설치하면 안 되나요?
```
가능하지만, 문제가 생깁니다:
• 서버 A와 서버 B의 환경이 다르면 동작이 달라짐
• "내 컴퓨터에서는 되는데..." 문제
• 새 서버 세팅할 때마다 수작업

Docker를 쓰면:
• 어떤 서버에서 실행해도 동일하게 동작
• 새 서버 세팅이 한 줄 명령어 (docker run)
```

### Q4. EC2, ECR, Triton의 관계가 뭔가요? 서버가 몇 개인가요?

**"서버"를 두 가지로 구분해야 합니다:**

| 구분 | 이름 | 역할 | 비유 |
|------|------|------|------|
| **하드웨어(컴퓨터)** | **EC2** | 물리적 컴퓨터(CPU, GPU, 메모리) | **건물** |
| **소프트웨어(프로그램)** | **Triton** | 그 위에서 돌아가는 AI 추론 프로그램 | **건물 안의 식당** |
| **저장소** | **ECR** | Docker 이미지 보관 창고 | **냉동 창고** |

```
오늘 사용한 것:
• EC2 2개 = 컴퓨터 2대 (개발용, 프로덕션용)
• ECR 1개 = 이미지 저장소 (창고)
• Triton 1개 = EC2 #2 안에서 돌아가는 프로그램

┌─────────────────────────────────────────────────────────────┐
│  EC2 #1 (컴퓨터)     ECR (창고)      EC2 #2 (컴퓨터)        │
│  ┌──────────┐      ┌─────────┐     ┌──────────────┐        │
│  │ 빌드     │─Push→│ 이미지  │─Pull→│ ┌──────────┐│        │
│  └──────────┘      └─────────┘     │ │ Triton   ││ ← 여기서 실행!
│                                    │ │ (서버SW) ││        │
│                                    │ └──────────┘│        │
│                                    └──────────────┘        │
└─────────────────────────────────────────────────────────────┘
```

### Q5. Triton과 vLLM의 차이는 뭔가요?

둘 다 "AI 모델을 서빙하는 소프트웨어"지만 **전문 분야가 다릅니다.**

| | **Triton (오늘 사용)** | **vLLM** |
|---|---|---|
| **전문 분야** | **범용 AI** (이미지, 추천, 음성 등) | **LLM 전용** (ChatGPT 같은 모델) |
| **지원 모델** | ONNX, TensorRT, PyTorch 등 다양 | Llama, Mistral 같은 LLM |
| **비유** | **종합병원** | **안과 전문병원** |

### Q6. FastAPI는 뭔가요? 왜 여기서 안 썼나요?

**FastAPI = Python으로 웹 API 서버를 직접 만드는 도구**

```python
# FastAPI 예시
from fastapi import FastAPI
app = FastAPI()

@app.post("/predict")
def predict(data):
    return my_model(data)
```

**왜 오늘 안 썼나요?**
→ Triton이 이미 그 역할을 해주기 때문입니다!

| | **FastAPI 직접 구현** | **Triton 사용** |
|---|---|---|
| **장점** | 자유도 높음 | 이미 최적화 완료 |
| **단점** | 동시 요청, GPU 관리 직접 구현 | 커스터마이징 제한적 |
| **언제?** | 복잡한 전/후처리 필요 시 | 표준 모델 빠르게 서빙 시 |

# 1. 이 실습이 왜 중요한가?

## 🤔 AI 모델을 만들었다. 그 다음은?

여러분이 PyTorch나 TensorFlow로 AI 모델을 학습시켰다고 가정해 봅시다.

```python
# 학습 완료된 모델
model = train_my_awesome_model()
model.save("my_model.onnx")
```

**문제**: 이 모델을 어떻게 실제 서비스에서 쓸 수 있을까요?

### ❌ 잘못된 방법: 노트북에서 직접 실행
```python
# main.py
result = model.predict(user_input)  # 🔥 이건 서비스가 아니다!
```

이렇게 하면:
- 동시에 1000명이 요청하면? → 서버 터짐
- 24시간 가동해야 하는데? → 내 컴퓨터를 끄면 안 됨
- 다른 나라에서 접속하면? → 느림

### ✅ 올바른 방법: 추론 서버 배포
```
사용자 → 인터넷 → 클라우드 서버(AI 모델) → 결과 반환
```

오늘 실습은 바로 이 **"AI 모델을 클라우드에서 서비스하는 방법"**을 배우는 것입니다.

## 🎯 이 실습에서 배우는 것

| 기술 | 왜 배우나? |
|------|----------|
| **Docker** | AI 환경을 패키징해서 어디서든 동일하게 실행 |
| **Triton** | AI 추론을 전문적으로 처리하는 서버 |
| **AWS EC2** | 클라우드에서 GPU 서버 사용 |
| **AWS ECR** | Docker 이미지를 클라우드에 저장 |
| **GPU 연동** | AI 추론 속도를 수십 배 향상 |

# 3. 핵심 기술 개념 완전 정복

## 3.1 Docker란 무엇인가?

### 🤔 Docker가 해결하는 문제

"**내 컴퓨터에서는 되는데...**" 문제를 해결합니다.

```
개발자 A: Python 3.8 + TensorFlow 2.5
개발자 B: Python 3.10 + TensorFlow 2.11
서버:      Python 3.9 + ???

→ 서로 다른 환경에서 동일하게 실행되지 않음!
```

### 📦 Docker의 해결책: 컨테이너

**컨테이너** = 애플리케이션 + 필요한 모든 것(라이브러리, 설정, 파일)을 하나로 묶음

```
┌─────────────────────────────┐
│        Docker 컨테이너       │
│  ┌───────────────────────┐  │
│  │   내 애플리케이션      │  │
│  ├───────────────────────┤  │
│  │   Python 3.8          │  │
│  │   TensorFlow 2.5      │  │
│  │   CUDA 11.8           │  │
│  │   필요한 모든 파일     │  │
│  └───────────────────────┘  │
└─────────────────────────────┘
        ↓
   어떤 서버에서 실행해도
   항상 동일하게 작동!
```

### 🔧 Docker 핵심 용어

| 용어 | 설명 | 비유 |
|------|------|------|
| **이미지 (Image)** | 컨테이너의 설계도, 읽기 전용 | 붕어빵 틀 |
| **컨테이너 (Container)** | 이미지로 만든 실행 중인 인스턴스 | 붕어빵 |
| **Dockerfile** | 이미지를 만드는 레시피 | 붕어빵 레시피 |
| **레지스트리 (Registry)** | 이미지 저장소 | 붕어빵 틀 보관 창고 |

### 🔄 Docker 기본 명령어 흐름

```bash
# 1. 이미지 다운로드 (Registry → 로컬)
docker pull nginx

# 2. 컨테이너 실행 (이미지 → 컨테이너)
docker run -d --name my-web nginx

# 3. 실행 중인 컨테이너 확인
docker ps

# 4. 컨테이너 중지
docker stop my-web

# 5. 컨테이너 삭제
docker rm my-web
```

## 3.4 Triton Inference Server

### 🤔 왜 Flask/FastAPI가 아니라 Triton인가?

```python
# 단순한 방법 (Flask)
from flask import Flask
app = Flask(__name__)

@app.route("/predict")
def predict():
    result = model(input)  # 🔥 동시 요청 처리 어려움
    return result          # 🔥 배치 최적화 없음
                           # 🔥 GPU 메모리 관리 수동
```

Triton은 이런 문제들을 **전문적으로** 해결합니다.

### 🚀 Triton의 핵심 기능

| 기능 | 설명 |
|------|------|
| **동적 배칭** | 여러 요청을 모아서 한 번에 처리 (처리량 ↑) |
| **멀티 모델 서빙** | 하나의 서버에서 여러 모델 동시 운영 |
| **멀티 프레임워크** | ONNX, TensorRT, PyTorch, TensorFlow 모두 지원 |
| **모델 버전 관리** | 여러 버전의 모델을 동시 운영 |
| **메트릭 제공** | Prometheus 형식의 모니터링 지표 |
| **gRPC 지원** | HTTP보다 빠른 통신 프로토콜 |

### 📊 Triton의 포트 구성 (8000, 8001, 8002)

Triton 서버는 용도에 따라 **세 종류의 문(Port)**을 열어둡니다.

| 포트 번호 | 이름 | 주요 용도 | 비유 |
|:---:|:---:|---|---|
| **8000** | **HTTP / REST** | 일반적인 웹 요청, 간단한 테스트, Health Check | **일반 우편** (사용하기 편하지만 조금 느림) |
| **8001** | **gRPC** | 고속 데이터 전송, 대량의 추론 요청 (실무용) | **전용 팩스** (빠르고 효율적임) |
| **8002** | **Metrics** | 서버의 상태(CPU/GPU 사용량, 에러율) 모니터링 | **CCTV / 계량기** (운영 상태 확인) |

#### 1️⃣ Port 8000 (HTTP/REST)
- **특징**: 웹 브라우저나 `curl` 명령어로 가장 쉽게 써볼 수 있습니다.
- **언제 쓰나?**: 서버가 잘 켜졌인지(`health/ready`) 확인하거나, 간단하게 한두 번 테스트할 때 씁니다.
- **예시**: `curl http://IP:8000/v2/health/ready`

#### 2️⃣ Port 8001 (gRPC)
- **특징**: 데이터를 '압축'해서 주고받기 때문에 HTTP보다 속도가 훨씬 빠릅니다.
- **언제 쓰나?**: 실제 서비스(App, Web)가 Triton과 통신할 때, 혹은 1초에 수천 번씩 추론해야 할 때 씁니다.
- **예시**: Python 클라이언트에서 `tritonclient.grpc` 모듈로 접속

#### 3️⃣ Port 8002 (Metrics)
- **특징**: 서버의 "건강 점수"를 텍스트로 보여줍니다.
- **언제 쓰나?**: "GPU 메모리가 얼마나 남았지?", "지금까지 총 몇 번 추론했지?" 같은 통계를 낼 때 씁니다.
- **예시**: `curl http://IP:8002/metrics`

# 4. Step 1: EC2 접속 - 완전 해부 `EC2 #1`

## 🎯 이 단계에서 하는 것

**AWS 클라우드에 있는 EC2 서버에 원격으로 접속**합니다.

```
내 컴퓨터 (Windows/Mac) ───SSH 터널───► EC2 서버 (Ubuntu + GPU)
```

## ❓ 왜 필요한가?

| 상황 | 설명 |
|------|------|
| 내 컴퓨터로 서버 돌리기? | 24시간 켜놔야 함, 전기세, 인터넷 품질 문제 |
| 내 컴퓨터에 GPU 없음 | AI 추론이 엄청 느림 |
| 서비스 확장 어려움 | 사용자 늘어나면 서버 추가 불가 |

**해결책**: AWS에서 GPU 달린 서버(EC2)를 빌려서, 거기에 원격 접속해서 작업합니다.

## 🔐 SSH란?

**S**ecure **SH**ell - 원격 컴퓨터에 안전하게 접속하는 프로토콜

```
내 컴퓨터                 인터넷                  EC2
┌─────────┐              ════════              ┌─────────┐
│         │  ──── 암호화된 터널 ────►         │         │
│  SSH    │      (22번 포트)                  │  SSHD   │
│ 클라이언트│  ◄─────────────────────          │  서버   │
└─────────┘                                   └─────────┘
```

## 📝 명령어 완전 해부

```bash
ssh -i "mykey.pem" ubuntu@3.123.45.67
```

| 부분 | 설명 | 상세 |
|------|------|------|
| `ssh` | SSH 클라이언트 프로그램 | Windows: OpenSSH 또는 PuTTY |
| `-i "mykey.pem"` | Identity file (신분증) | EC2 생성 시 다운받은 키 파일 |
| `ubuntu` | 접속할 사용자 이름 | Ubuntu AMI의 기본 사용자 |
| `@` | 구분자 | "사용자@서버" 형식 |
| `3.123.45.67` | EC2의 퍼블릭 IP | AWS 콘솔에서 확인 |

## 🔑 PEM 키 파일의 원리

```
                  비대칭 암호화
                  
┌─────────────────┐              ┌─────────────────┐
│  공개 키        │              │  개인 키        │
│  (Public Key)   │              │  (Private Key)  │
│                 │              │                 │
│  EC2에 저장됨   │     ↔       │  내가 가지고 있음│
│                 │   암호화     │  (.pem 파일)    │
└─────────────────┘   복호화     └─────────────────┘
```

1. EC2를 만들 때 키 페어를 생성
2. 공개 키는 EC2 안에 저장됨 (`~/.ssh/authorized_keys`)
3. 개인 키(.pem)는 딱 한 번만 다운로드 가능
4. 접속 시 개인 키로 "내가 주인이다"를 증명

## ⚠️ 왜 파일 권한이 중요한가?

```bash
# Linux/Mac에서
chmod 400 mykey.pem  # 나만 읽기 가능
```

SSH는 보안을 위해 **"다른 사람이 읽을 수 있는 키 파일"을 거부**합니다.

```
권한 644 (rw-r--r--): 다른 사람도 읽기 가능 → SSH 거부!
권한 400 (r--------): 나만 읽기 가능 → SSH 허용
```

### 💡 WSL 사용자 필수! 키 파일 복사

WSL(Windows Subsystem for Linux)에서 `/mnt/c/...` 경로(윈도우 폴더)에 있는 키 파일은 `chmod`가 적용되지 않습니다!

**에러 메시지:**
```
Permissions 0555 for 'q11-key.pem' are too open.
Load key "q11-key.pem": bad permissions
Permission denied (publickey).
```

**해결 방법 (필수!):**
```bash
# 1. 키 파일을 WSL 홈 디렉토리로 복사
cp "q11-key.pem" ~/

# 2. 복사한 파일에 권한 설정 (이제는 잘 됩니다!)
chmod 400 ~/q11-key.pem

# 3. 새로운 경로의 키로 접속
ssh -i ~/q11-key.pem ubuntu@<EC2-IP>
```

> 💡 **왜 이래야 하나요?**
> - `/mnt/c/` 경로는 윈도우 파일 시스템(NTFS)을 리눅스에서 보는 것
> - NTFS는 리눅스 권한 체계(chmod)를 완벽히 지원하지 않음
> - `~/` 경로는 진짜 리눅스 파일 시스템(ext4)이라 chmod가 정상 작동

# 5. Step 2: Docker 환경 구축 - 완전 해부

## 🎯 이 단계에서 하는 것

**Docker를 설치하고, 현재 사용자가 Docker 명령어를 쓸 수 있게** 설정합니다.

```
EC2 서버에 Docker Engine 설치
     ↓
ubuntu 사용자를 docker 그룹에 추가
     ↓
sudo 없이 docker 명령어 사용 가능
```

## ❓ 왜 필요한가?

| 질문 | 답변 |
|------|------|
| Docker 왜 쓰나? | Triton 서버를 컨테이너로 실행하기 위해 |
| 직접 설치하면 안 되나? | 가능하지만, 의존성 지옥에 빠질 수 있음 |
| 다른 서버에 배포하려면? | Docker 이미지 하나로 어디서든 동일하게 실행 |

**Docker가 없으면**: Triton 실행에 필요한 수십 개의 라이브러리를 일일이 설치해야 합니다.
**Docker가 있으면**: `docker run` 한 줄로 모든 환경이 갖춰진 상태에서 실행됩니다.

## 📥 Docker 설치 스크립트

```bash
curl -fsSL https://get.docker.com -o get-docker.sh
sudo sh get-docker.sh
```

### 명령어 분해

**첫 번째 줄: 설치 스크립트 다운로드**

| 옵션 | 의미 | 상세 |
|------|------|------|
| `curl` | 데이터 전송 도구 | URL에서 파일 다운로드 |
| `-f` | fail silently | HTTP 에러 시 조용히 실패 |
| `-s` | silent | 진행 표시 숨김 |
| `-S` | show error | 에러는 표시 |
| `-L` | follow redirects | 리다이렉트 따라감 |
| `-o 파일명` | output | 다운받을 파일 이름 지정 |

**두 번째 줄: 스크립트 실행**

| 부분 | 의미 |
|------|------|
| `sudo` | 관리자 권한으로 (Super User DO) |
| `sh` | 셸 스크립트 실행기 |
| `get-docker.sh` | 실행할 스크립트 |

## 👤 Docker 권한 설정

```bash
sudo usermod -aG docker $USER
newgrp docker
```

### 왜 필요한가?

Docker 명령을 실행하려면 `/var/run/docker.sock` 소켓에 접근해야 합니다.
이 소켓의 기본 권한은 `root` 또는 `docker` 그룹만 접근 가능합니다.

```bash
# docker.sock 권한 확인
ls -la /var/run/docker.sock
# 출력: srw-rw---- 1 root docker ...
#       ↑ root 또는 docker 그룹만 읽기/쓰기 가능
```

### 명령어 분해

```bash
sudo usermod -aG docker $USER
```

| 부분 | 의미 |
|------|------|
| `usermod` | 사용자 설정 수정 (user modify) |
| `-a` | append (기존 그룹에 추가) |
| `-G docker` | docker 그룹에 |
| `$USER` | 현재 사용자 (ubuntu) |

```bash
newgrp docker
```

| 부분 | 의미 |
|------|------|
| `newgrp` | 새 그룹으로 셸 시작 (재로그인 없이 그룹 적용) |
| `docker` | 적용할 그룹 이름 |

# 6. Step 3: NVIDIA GPU 연동 - 완전 해부

## 🎯 이 단계에서 하는 것

**Docker 컨테이너 안에서 GPU를 사용할 수 있도록** NVIDIA Container Toolkit을 설치합니다.

```
Docker 컨테이너
      │
      ╳ 기본적으로 GPU 인식 못 함
      │
      ▼
NVIDIA Container Toolkit 설치
      │
      ▼
      ✓ --gpus 옵션으로 GPU 연결 가능!
```

## ❓ 왜 필요한가?

| 상황 | CPU만 | GPU 사용 |
|------|-------|----------|
| 이미지 1장 추론 | 100ms | **5ms** |
| 배치 32장 추론 | 3200ms | **20ms** |
| 동시 100명 요청 | 서버 터짐 | 문제없음 |

**GPU 없이 AI 추론**은 가능하지만, 실서비스에서는 **응답 속도가 너무 느려서** 사용자 불만이 생깁니다.
Triton이 GPU를 쓰려면 Docker가 GPU를 인식해야 하고, 그걸 가능하게 하는 게 이 Toolkit입니다.

## 📦 NVIDIA Container Toolkit 설치

### 1단계: GPG 키 등록

```bash
curl -fsSL https://nvidia.github.io/libnvidia-container/gpgkey | \
  sudo gpg --dearmor -o /usr/share/keyrings/nvidia-container-toolkit-keyring.gpg
```

**이게 뭐하는 거야?**

패키지 저장소의 **진짜 NVIDIA에서 온 것인지 확인**하기 위한 공개 키를 등록합니다.

```
NVIDIA 서버                        내 서버
┌─────────────┐                   ┌─────────────┐
│ 패키지 서명 │ ───► 인터넷 ───► │ 서명 검증   │
│ (개인 키)   │                   │ (공개 키)   │
└─────────────┘                   └─────────────┘
              "이 패키지는 진짜 NVIDIA가 만든 거야"
```

| 부분 | 의미 |
|------|------|
| `gpg --dearmor` | ASCII 키를 바이너리로 변환 |
| `-o /usr/share/keyrings/...` | 키 저장 위치 |

### 2단계: 저장소 등록

```bash
curl -s -L https://nvidia.github.io/libnvidia-container/stable/deb/nvidia-container-toolkit.list | \
  sed 's#deb https://#deb [signed-by=/usr/share/keyrings/nvidia-container-toolkit-keyring.gpg] https://#g' | \
  sudo tee /etc/apt/sources.list.d/nvidia-container-toolkit.list
```

**이게 뭐하는 거야?**

"NVIDIA 패키지를 어디서 다운받을지"를 시스템에 알려줍니다.

1. `curl`: NVIDIA의 저장소 주소 목록을 가져옴
2. `sed`: 그 목록에 "서명 검증용 키 위치"를 추가
3. `tee`: 결과를 apt 설정 폴더에 저장

### 3단계: 설치 및 적용

```bash
sudo apt-get update                          # 저장소 목록 갱신
sudo apt-get install -y nvidia-container-toolkit  # 설치
sudo nvidia-ctk runtime configure --runtime=docker  # Docker에 연결
sudo systemctl restart docker                # Docker 재시작
```

# 7. Step 4: Model Repository 구성 - 완전 해부

## 🎯 이 단계에서 하는 것

**AI 모델 파일(.onnx)을 Triton이 인식할 수 있는 폴더 구조로 배치**하고,
**모델의 입출력 형태를 정의하는 config.pbtxt 파일을 작성**합니다.

```
우리가 할 일:

1. 폴더 생성: model_repository/simple_onnx/1/
2. 모델 다운로드: model.onnx (MNIST 숫자 인식)
3. 설정 파일 작성: config.pbtxt
```

## ❓ 왜 필요한가?

**Triton은 아무 폴더에 있는 모델을 그냥 못 읽습니다!**

정해진 규칙을 따라야만 모델을 로딩합니다:

| 규칙 | 설명 |
|------|------|
| 폴더 이름 = 모델 이름 | `simple_onnx/` 폴더면 API에서 `simple_onnx`로 호출 |
| 버전 폴더 필수 | `1/`, `2/` 등 숫자 폴더 안에 모델 파일 |
| config.pbtxt 필수 | 입력/출력 텐서의 이름, 타입, 크기 정의 |
| 파일명 규칙 | ONNX는 `model.onnx`, TensorRT는 `model.plan` |

이 규칙을 어기면 Triton 시작 시 **"모델 로딩 실패"** 에러가 납니다.

## 📁 Triton Model Repository란?

Triton은 **특정 폴더 구조**에 모델이 있어야만 인식합니다.

```
model_repository/              ← 최상위 폴더 (이름 자유)
├── model_A/                   ← 모델 이름 (Triton에서 이 이름으로 호출)
│   ├── config.pbtxt          ← 모델 설정 (필수!)
│   ├── 1/                    ← 버전 1
│   │   └── model.onnx        ← 모델 파일
│   └── 2/                    ← 버전 2 (선택)
│       └── model.onnx
├── model_B/
│   ├── config.pbtxt
│   └── 1/
│       └── model.plan        ← TensorRT 모델
└── model_C/
    ├── config.pbtxt
    └── 1/
        └── model.graphdef    ← TensorFlow 모델
```

## 🔧 config.pbtxt 완전 해부

```protobuf
name: "simple_onnx"
platform: "onnxruntime_onnx"
max_batch_size: 0

input [
  {
    name: "Input3"
    data_type: TYPE_FP32
    dims: [ 1, 1, 28, 28 ]
  }
]

output [
  {
    name: "Plus214_Output_0"
    data_type: TYPE_FP32
    dims: [ 1, 10 ]
  }
]
```
